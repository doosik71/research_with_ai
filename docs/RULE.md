# 문서 작성 및 연구 작업 규칙

이 문서는 이 저장소에서 수행하는 문서 작성, 논문 조사, 논문 요약 작업의 기본 규칙을 정의한다.
별도 지시가 없는 한 모든 작업은 이 문서를 기준으로 수행한다.

## 1. 적용 범위와 우선순위

- 이 규칙은 저장소 전체에 적용한다.
- 사용자가 현재 작업에 대해 별도 지시를 내리면, 해당 지시는 `RULE.md` 보다 우선한다.
- 사용자가 "기억하라", "앞으로 이 규칙을 따르라"와 같이 지속 적용을 요구한 지침은 별도 언급이 없는 한 이 문서에 반영한다.

## 2. 파일 및 폴더 명명 규칙

- 모든 파일명과 폴더명은 영문 소문자를 사용한다.
- 공백은 사용하지 않고 반드시 언더스코어(`_`)로 대체한다.
- 특수문자는 가급적 사용하지 않는다. 꼭 필요하지 않다면 영문, 숫자, 언더스코어만 사용한다.
- 이 규칙은 Windows, Linux, macOS 환경에서의 호환성을 위한 기본 규칙이다.
- 예시:
  - `my folder` -> `my_folder`
  - `Energy Based Models.md` -> `energy_based_models.md`

## 3. 파일 형식과 인코딩

- 기본 문서 형식은 Markdown(`.md`)이다.
- 사용자가 명시적으로 요청한 경우에만 `.txt`, `.json` 등 다른 형식을 사용한다.
- 모든 텍스트 파일은 UTF-8 인코딩을 사용한다.
- 줄바꿈은 Unix 스타일 개행(`LF`)을 사용한다.
- 파일 끝에는 개행을 하나 둔다.

## 4. Markdown 작성 규칙

### 4.1 헤더 구조

- 헤더는 `#` 문법을 사용한다.
- 헤더 레벨은 논리적으로 한 단계씩 진행한다.
- 문서 구조가 길어질 경우 상위 섹션부터 하위 섹션 순서로 일관되게 구성한다.

### 4.2 링크, 이미지, 코드 블록

- 링크와 이미지 경로는 상대 경로를 사용한다.
- 링크 대상 파일도 동일한 파일명 규칙을 따른다.
- 코드 블록에는 가능한 한 언어 식별자를 명시한다.

예시:

```python
def hello():
    print("hello")
```

## 5. 수식 표기 규칙

- 문서 내 수식은 MathJax 표기법을 사용한다.
- 인라인 수식은 `$...$`를 사용한다.
- 블록 수식은 `$$...$$`를 사용한다.
- `\(...\)`, `\[...\]` 형태의 LaTeX 수식이 있다면 각각 `$...$`, `$$...$$`로 변환한다.

예시:

- `\(E = mc^2\)` -> `$E = mc^2$`
- `\[\int_{a}^{b} f(x)dx\]` -> `$$\int_{a}^{b} f(x)dx$$`

## 6. 문서 품질 검사

- Markdown 문서는 `markdownlint` 또는 동등한 도구로 검사한다.
- Lint 오류가 있으면 반드시 수정한다.
- 경고는 가능한 한 해소한다. 해소가 어렵다면 문서 안에 이유가 드러나도록 작성한다.
- 특히 논문 목록 문서와 요약 문서를 새로 만들거나 수정한 뒤에는 Lint 검사를 다시 수행한다.

## 7. 저장소 구조와 폴더 운영 규칙

이 절에서 말하는 문서 루트는 저장소 루트가 아니라 `docs` 폴더를 뜻한다.

### 7.1. 루트 인덱스 파일

- 문서 루트(`docs`)의 연구 주제별 작업은 해당 주제 폴더 안에서 수행한다.
- `docs` 폴더에는 반드시 `index.md`가 있어야 한다.
- `docs/index.md` 파일은 서브 폴더의 목록을 관리한다.
- 특정 연구 분야의 생성 작업 요청을 받으면 `docs` 아래에 해당 분야의 폴더를 생성하고 `docs/index.md` 파일을 갱신한다.
- 루트 인덱스 파일의 각 항목은 연구 주제명으로 정렬한다.
- 루트 인덱스 파일의 기본 예시:

```markdown
# Index

- [연구 주제명 1](sub_folder_1.md)
- [연구 주제명 2](sub_folder_2.md)
```

### 7.2. 서브 폴더의 인덱스 파일

- 연구 주제별 작업은 `docs` 아래의 해당 주제 폴더 안에서 수행한다.
- 각 연구 주제 폴더에는 반드시 `index.md`가 있어야 한다.
- 각 연구 주제 폴더에는 반드시 `metadata.json`이 있어야 한다.
- `index.md`는 해당 폴더의 문서 목록 역할을 하며, 새 문서를 추가할 때마다 함께 갱신한다.
- 특정 분야 작업 요청을 받으면 해당 분야의 폴더에서 자료 조사, 문서 생성, 목록 갱신까지 완료한다.
- 서브 폴더 인덱스 파일의 각 항목은 논문명으로 정렬한다.
- 서브 폴더 인덱스 파일의 기본 예시:

```markdown
# 연구 주제명

- [논문명 1](document_1.md)
- [논문명 2](document_2.md)
```

### 7.3. 서브 폴더 메타데이터 파일

- 각 연구 주제 폴더에는 `metadata.json` 파일을 둔다.
- `metadata.json`은 해당 폴더의 연구 주제명과 논문 검색용 키워드를 정의한다.
- `metadata.json`에는 최소한 아래 필드를 포함한다.
  - `title`
  - `keyword`
- `title`은 사람이 읽는 연구 주제명을 나타내는 문자열이다.
- `keyword`는 arXiv 등에서 해당 주제 논문을 검색할 때 사용할 문자열 배열이다.
- 검색 전략이 바뀌거나 더 적절한 검색 문자열이 발견되면 `keyword`를 갱신한다.
- 새 연구 분야 폴더를 생성할 때는 `index.md`와 함께 `metadata.json`도 반드시 생성한다.

예시:

```json
{
  "title": "Continual Learning",
  "keyword": [
    "continual learning"
  ]
}
```

## 8. 논문 조사 규칙

- 논문 조사는 기본적으로 arXiv를 우선 사용한다.
- 다른 출처가 필요하거나 더 적절한 경우에는 그 이유가 명확할 때만 예외적으로 사용한다.
- 조사한 논문 목록은 `docs` 아래 각 연구 분야 폴더의 `paper_list.jsonl`에 기록한다.
- `paper_list.jsonl`의 각 항목에는 아래 정보를 포함한다.
  - 저자명
  - 논문명
  - 발표 연도
  - 논문 URL
- 요약 보고서 파일명
- 발표자료 파일명
- 논문 제목과 저자명은 번역하지 않고 원문 그대로 유지한다.
- 새 논문 정보를 추가할 때 논문이 중복되지 않도록 관리한다.
- `summary`는 마크다운 상세 분석 보고서 파일명이며, 없으면 빈 문자열로 둔다.
- `slide`는 Marp 발표자료 파일명이며, 없으면 빈 문자열로 둔다.

예시:

```json
{"title":"Continual Learning with Elastic Weight Consolidation","author":"John Doe, Jane Smith","year":"2020","url":"https://arxiv.org/abs/2001.12345","summary":"continual_learning_with_elastic_weight_consolidation.md","slide":"continual_learning_with_elastic_weight_consolidation_slide.md"}
```

- 논문 검색에 대한 보다 상세한 규칙은 `RULE_SEARCH.md`의 문서에 명시된 내용을 따른다.

## 9. 논문 요약 작업 규칙

### 9.1 요청 해석

- 사용자가 논문 요약을 요청하면, 단순 요약이 아니라 상세 분석 보고서 작성 요청으로 해석한다.

### 9.2 기본 작업 순서

논문 요약 작업은 아래 순서를 기본으로 한다.

1. arXiv에서 대상 논문을 찾는다.
2. arXiv 페이지에서 PDF 링크를 확인한다.
3. PDF를 기반으로 본문 텍스트를 추출한다.
4. 초록, 서론, 방법, 실험, 결론 등 주요 섹션을 식별한다.
5. 상세 분석 보고서를 Markdown으로 작성한다.
6. 해당 연구 분야 폴더에 저장한다.
7. `index.md`에 링크를 추가한다.
8. 필요하면 `paper_list.jsonl`의 `summary`, `slide` 등 메타데이터를 갱신한다.
9. 마지막으로 Markdown lint를 수행한다.

### 9.3 보고서 필수 포함 항목

논문 요약 보고서에는 최소한 아래 내용을 포함한다.

1. 논문 메타데이터
2. 연구 배경 및 문제 정의
3. 핵심 기여
4. 방법론 요약
5. 실험 설정과 결과
6. 한계 및 향후 연구 가능성
7. 실무적 또는 연구적 인사이트

세부 기준:

- 메타데이터에는 저자, 제목, 발표 연도, arXiv ID를 포함한다.
- DOI가 확인 가능하면 함께 적는다.
- 방법론 설명에 수식이 필요하면 MathJax 형식을 사용한다.
- 가능하면 표, 그림, 캡션에서 확인되는 핵심 결과도 반영한다.
- 보고서는 원문을 읽지 않아도 전체 흐름을 이해할 수 있을 정도로 충분히 상세해야 한다.

### 9.4 언어 정책

- 보고서 본문은 한국어로 작성한다.
- 모델명, 알고리즘명, 논문 제목, 저자명, 수식, 고유 기술 용어는 영어 원문을 유지할 수 있다.

### 9.5 파일명과 저장 위치

- 요약 보고서 파일명은 논문의 영어 제목을 소문자와 언더스코어 형식으로 변환해 사용한다.
- 확장자는 `.md`를 사용한다.
- 파일은 `docs` 아래 해당 논문이 속한 연구 주제 폴더에 저장한다.
  - 예시:
    - `energy_based_models_for_continual_learning.md`

### 9.6 발표 자료

- 발표자료는 Marp 형식(<https://marp.app/>)을 따른다.
- 발표 자료의 파일명은 파일명의 뒤에 `_slide`라는 postfix를 추가한다.
  - 예시:
    - `energy_based_models_for_continual_learning_slide.md`
  
## 10. 커밋 메시지 규칙

- 문서 변경 커밋에는 `docs:` 접두사를 사용한다.
- 논문 요약 파일을 추가하거나 크게 수정할 때는 아래 형식을 우선 사용한다.
  - 예시:

```text
docs: add summary <paper_title>
```

- 두 편 이상의 논문 정보가 추가 또는 업데이트되었으면 아래 예시와 같이 커밋 메시지를 작성할 수 있다.
  - 예시:

```text
docs: update <research area>

- add <paper_title_1>
- update <paper_title_2>
```

## 11. 작업 체크리스트

문서 또는 논문 요약 작업을 마치기 전에 아래 항목을 확인한다.

- 파일명과 폴더명이 규칙을 따르는가
- Markdown 형식과 헤더 구조가 일관적인가
- 수식 표기가 MathJax 형식으로 정리되었는가
- 새 문서가 올바른 폴더에 저장되었는가
- `metadata.json`이 존재하고 내용이 현재 연구 주제와 일치하는가
- `index.md`가 갱신되었는가
- 필요 시 `paper_list.jsonl`이 갱신되었는가
- `markdownlint` 검사를 통과했는가
